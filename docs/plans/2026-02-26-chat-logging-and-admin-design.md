# Chat Logging & Admin Analytics — Design

**Date:** 2026-02-26
**Status:** Approved, pending implementation plan

---

## Overview

Two parallel workstreams:

1. **nakom.is changes** — log every chat request to DynamoDB; monitor table growth; alert via email + SMS.
2. **New `nakom-admin` project** — admin web app at `admin.nakom.is` with an analytics dashboard: on-demand RDS, vector similarity graph, topic clustering, tool usage, spam detection, and cheap IP blocking via CloudFront Functions.

---

## 1. nakom.is Changes

### 1.1 DynamoDB Chat Logging Table

Added to `ChatStack`. Table name: `nakomis-chat-logs`.

**Key design:**

| Attribute | Type | Role |
|-----------|------|------|
| `logType` | String | Partition key. Value `"CVCHAT"` for nakom.is chat. Future agents (e.g. `"ADMINCHAT"`) use different values, keeping partitions warm and the table reusable. |
| `sk` | String | Sort key. Format: `{ISO-8601-timestamp}#{requestId}` — sortable and unique. |

**Item attributes:**

| Attribute | Type | Notes |
|-----------|------|-------|
| `conversationId` | String | UUID generated by the frontend on widget open. Groups turns of the same conversation. Not a nonce — it's a correlation/session ID with no security guarantee; spoofing it affects analytics only. |
| `ip` | String | First value from `X-Forwarded-For` header. CloudFront places the real viewer IP first. |
| `userAgent` | String | From `user-agent` request header. |
| `country` | String | From `CloudFront-Viewer-Country` header (added automatically by CF, no cost). |
| `userMessage` | String | The user's message in this turn. Not just the first message — each invocation is one turn, so each record captures the message that triggered it. |
| `messageCount` | Number | Number of messages in the conversation history at the time of this request (depth indicator). |
| `toolsCalled` | StringSet | Tool names invoked by the agent during this turn (e.g. `get_cv`, `read_repo_file`). |
| `inputTokens` | Number | From LangChain `handleLLMEnd` callback. |
| `outputTokens` | Number | From LangChain `handleLLMEnd` callback. |
| `durationMs` | Number | Wall-clock time for the full request. |
| `rateLimited` | Boolean | Whether this request hit the daily cap. |
| `ttl` | Number | Unix epoch timestamp, 1 year from creation. DynamoDB TTL attribute. |

**Table config:** PAY_PER_REQUEST billing. TTL enabled on `ttl` attribute.

**Import cursor:** SSM Parameter Store at `/nakom.is/analytics/last-imported-timestamp` (one per `logType` — so `/nakom.is/analytics/CVCHAT/last-imported-timestamp`). Set to `"1970-01-01T00:00:00Z"` on first deploy. The import Lambda queries `PK="CVCHAT" AND SK > {lastImported}` and updates the parameter on success. Single-user admin tool — no concurrency concern.

### 1.2 Stream Handler Changes

After `executor.invoke()` returns, a DynamoDB `PutItem` is fired **asynchronously** (not awaited before streaming `done`). If it fails, the error is logged to CloudWatch and the request continues — analytics data loss is acceptable; blocking a visitor's chat is not.

`SSECallbackHandler` is extended to also implement `handleLLMEnd`, which gives `response.llmOutput.tokenUsage` — input and output token counts extracted here and passed to the logging call.

The regular (non-streaming) `handler.ts` gets the same logging addition.

### 1.3 Frontend Change

The chat widget generates a `conversationId` UUID (v4) when it opens. This is included in every request body alongside `messages`. The Lambda reads it and logs it as `conversationId`.

### 1.4 Monitoring Lambda

New Lambda in `ChatStack` (or a new `MonitoringStack` in the same account):

- **Trigger:** EventBridge scheduled rule, daily at 08:00 UTC.
- **Logic:**
  1. Query DDB for records added in the last 24h (SK prefix scan on today's date).
  2. Read approximate table item count from `DescribeTable`.
  3. Compare against configurable thresholds (env vars: `DAILY_REQUEST_ALERT_THRESHOLD`, `TOTAL_ITEM_ALERT_THRESHOLD`).
  4. Send SES email (always, if thresholds exceeded).
  5. Send SNS SMS only if no SMS has been sent today. SMS-sent-today is tracked as a DynamoDB item with a 24h TTL (same `nakomis-chat-logs` table, `logType="SMS_SENT"`, avoids an extra resource). Capped at 1 SMS per day.
- **SMS:** SNS topic with a UK mobile number subscription. ~£0.05/message.

### 2.2 Infrastructure Stacks

**Deploy order:**
```
CertificateStack (us-east-1) → CognitoStack → CloudfrontStack → AnalyticsStack → ApiStack
```

All stacks use `AWS_PROFILE=nakom.is-admin` (same account as nakom.is).

**Route53:** The `nakom.is` hosted zone already exists. `CloudfrontStack` uses `HostedZone.fromLookup()` to add the `admin.nakom.is` A record — no changes needed in the nakom.is project's Route53 stack.

**Cognito:** Single-user User Pool. Username/password + TOTP MFA. No federated login needed. Identity Pool for AWS SDK access from the browser (same pattern as sandbox).

**RDS:** `db.t4g.micro` PostgreSQL 16 with `pgvector` extension. Stopped by default. Lives in a private VPC subnet. The `rds-control` Lambda starts/stops it on demand; starting takes ~3 minutes and the UI shows a polling status indicator. AWS will auto-start a stopped RDS instance after 7 days — a CloudWatch alarm fires if the instance has been running >12h unexpectedly, and the monitoring Lambda stops it.

**Snapshots:** Manual snapshots only (automated backups are unreliable for an instance that is usually stopped). The admin page triggers a snapshot before stopping. Keep 4 most recent, delete oldest on each backup cycle.

### 2.3 Analytics Page — Features

#### RDS Control Panel

```
[ ▶ Start RDS ]    Status: ● Stopped
[ ⟳ Restore last snapshot ]    2026-02-25 14:32
[ ↓ Import 47 new records ]    Last import: 2026-02-24 09:15
[ ● Backup & Stop ]            Snapshots: 4 kept (oldest auto-pruned)
```

All buttons call API Lambda endpoints. RDS start/stop uses the AWS SDK `startDBInstance` / `stopDBInstance`. Status is polled every 5s while transitioning.

#### Semantic Similarity Graph

D3.js force-directed graph. Each node is one chat turn, sized by message length, coloured by cluster. Edges connect pairs with cosine similarity above a threshold (tunable slider). Hovering a node shows the `userMessage`. Cluster labels are generated by the query Lambda (calls Claude Haiku on the 5 nearest messages to each cluster centroid → returns a short label like "Kubernetes experience" or "BootBoots spray detection").

Immediately surfaces spam patterns: a tight cluster of 50 nodes all from different IPs = coordinated prompt injection attempts or a viral question.

#### Topic Radar

Radar/spider chart of the AI-generated cluster labels by volume. Answers: what do visitors actually want to know? What skills are they asking about? Which projects interest them?

#### Tool Usage Bar Chart

`SELECT unnest(tools_called), count(*) FROM chat_logs GROUP BY 1 ORDER BY 2 DESC`

Shows which tools get used and which never fire. Low-use tools might be redundant. Conversations where `userMessage` contains interest signals but no tool was called (or the response contains "I don't have information about") are flagged as **missing tool candidates**.

#### IP Activity Heatmap

Calendar-style heatmap, one row per IP address, columns = days. Heavy spammer IPs show as a solid bar. Includes a "Conversations" drill-down: click an IP to see all their turns grouped by `conversationId`.

#### Bad Actor Log Mining

Manual trigger from the admin page. The user selects a time window (last week / last month) and clicks **Find bad actors**. A Lambda:

1. Lists CloudFront access log files from S3 for the selected period.
2. Downloads and parses the TSV format.
3. Aggregates by IP: total requests, unique paths, error rate, % of requests to non-existent paths.
4. Returns ranked list of suspicious IPs with flags:
   - `SCANNER` — >30% requests to paths like `/.env`, `/wp-admin`, `/.git`, `/phpmyadmin`
   - `RATE_LIMIT_ABUSER` — appears in DDB with `rateLimited=true` multiple times
   - `HIGH_VOLUME` — top 1% by request count

Each result has a **[+ Block]** button.

> **Prerequisite:** CloudFront access logging must be enabled, writing to the `nakom.is-static` bucket (or a dedicated logging bucket) with a `cf-logs/` prefix.

#### Blocklist Management

The blocklist is a JSON array stored in SSM Parameter Store at `/nakom.is/blocked-ips`:

```json
[
  { "ip": "203.0.113.42", "blockedAt": "2026-02-26T10:15:00Z", "reason": "vuln scan" },
  { "ip": "198.51.100.7", "blockedAt": "2026-02-26T10:16:00Z", "reason": "rate limit spam" }
]
```

**FIFO pruning:** when the serialised list approaches 3.5KB (SSM standard parameter limit is 4KB), the Lambda removes the oldest entries first before writing back. The admin UI displays the list in reverse-chronological order (newest at top).

**Blocking mechanism:** The existing CloudFront Function on the `nakom.is` distribution (currently handles `/social` → `/` rewriting) is extended to also block listed IPs. The blocked IP list is **baked directly into the function code** at deploy time — CloudFront Functions cannot make external calls, so the list must be embedded. The `blocklist` Lambda:

1. Reads the current list from SSM.
2. Renders a new CloudFront Function source (TypeScript template with the IP array inlined).
3. Calls the CloudFront API to update and publish the function.
4. Update propagates globally in ~10 seconds.

Cost: CloudFront Functions at $0.10/million invocations — effectively free at nakom.is volumes. No WAF required.

## 3. Draw.io Git Hook

`.githooks/pre-commit` detects modified `*.drawio` files and exports them to SVG using the draw.io CLI (`draw.io --export --format svg --output docs/diagrams/`). The generated SVGs are staged and included in the commit. Requires draw.io desktop app installed locally.

To activate: `git config core.hooksPath .githooks` (documented in README).

## 4. What Stays in nakom.is vs What Goes in nakom-admin

| Concern | Project |
|---------|---------|
| `nakomis-chat-logs` DDB table | nakom.is / ChatStack |
| Logging writes in stream-handler + handler | nakom.is / ChatStack |
| Monitoring Lambda + EventBridge rule | nakom.is / ChatStack (or new MonitoringStack) |
| SNS topic + SES alert | nakom.is |
| CloudFront Function (blocking + redirects) | nakom.is / CloudfrontStack |
| `/nakom.is/blocked-ips` SSM parameter | nakom.is account (written by nakom-admin Lambda) |
| RDS PostgreSQL + pgvector | nakom-admin / AnalyticsStack |
| Import Lambda (DDB → RDS) | nakom-admin / ApiStack |
| Query Lambda (analytics) | nakom-admin / ApiStack |
| CF log miner Lambda | nakom-admin / ApiStack |
| Blocklist Lambda (SSM + CF Function redeploy) | nakom-admin / ApiStack |
| Admin React app | nakom-admin / web |
| Cognito (admin auth) | nakom-admin / CognitoStack |

The nakom-admin Lambdas are granted IAM access to the nakom.is account resources they need (same account, straightforward IAM grants): read on `nakomis-chat-logs`, read/write on `/nakom.is/blocked-ips` SSM, `cloudfront:UpdateFunction` + `cloudfront:PublishFunction`, and `s3:GetObject` on the CloudFront logs prefix.
